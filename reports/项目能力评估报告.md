# doc-textify 项目能力评估报告

## 1. 关键结论

当前主分支代码没有回退。`current_red_cluster` 的 28.83% 是旧输出目录，不代表当前源码能力。

用当前源码重新运行同一张图片，结果是：

- 输出目录：`test_outputs\current_source_verify`
- 综合得分：97.84%
- required_terms：100%
- panel_layout：100%
- chart_data：94.59%
- usable_confidence：100%

因此，三项关键能力已经在当前主线中：

1. 中文 OCR 后处理。
2. 红色竖线与红色散点提取。
3. 带显式误差范围的深度读数评测。

## 2. 为什么 `current_red_cluster` 只有 28.83%

`current_red_cluster` 是早期输出，不是当前源码重新生成的结果。复核如下：

| 输出目录 | 总分 | required_terms | panel_layout | chart_data | 说明 |
| --- | ---: | ---: | ---: | ---: | --- |
| `current_red_cluster` | 28.83% | 16.67% | 16.67% | 5.41% | 旧输出，缺少后处理、散点、误差字段 |
| `current_source_verify` | 97.84% | 100% | 100% | 94.59% | 当前源码重新生成 |

两个目录的结构差异：

| 项目 | current_red_cluster | current_source_verify |
| --- | ---: | ---: |
| block 数 | 10 | 10 |
| chart_data 总数 | 12 | 34 |
| interval | 12 | 12 |
| point | 0 | 22 |
| depth_tolerance | 无 | 有 |
| OCR 关键文本 | `/m 深度`、`真实 类 别` | `深度/m`、`真实类别`、`预测类别` |

结论：28.83% 是旧产物残留导致的误判，不是当前代码能力。

## 3. 当前主线已包含的关键实现

### 3.1 中文 OCR 后处理

位置：`doc_textify/extractors.py`

核心功能：

- 合并中文字符之间的异常空格。
- 修正 `/m 深度`、`深度 /m` 为 `深度/m`。
- 修正 `真实 类别`、`预测 类别`。
- 修正 `钼孔` 到 `钻孔`。

### 3.2 红色图表元素提取

位置：`doc_textify/chart.py`

核心功能：

- HSV 红色掩码。
- 竖线和散点分离。
- 红色竖线按 x 坐标聚类。
- 散点按最近类别中心归类。
- 输出 interval 和 point 类型的 `chart_data`。

### 3.3 显式误差范围

位置：`doc_textify/chart.py`、`doc_textify/evaluation.py`、`doc_textify/renderers.py`

核心功能：

- interval 输出 `depth_tolerance`。
- point 输出 `depth_tolerance`。
- 评测器识别 `depth_tolerance`，避免把拍照图表读数当成无限精确值。
- Markdown/TXT/LLM 输出中显示 `+/-` 误差范围。

## 4. 与视觉大模型的关系

`doc-textify` 不替代通用视觉大模型。它适合成为文本大模型的“文档/图表视觉预处理器”。

适合场景：

- PDF、扫描件、截图、表格、工程图表。
- 需要低 token 成本。
- 需要可复查、可评测、可 JSON 化的视觉事实。
- 下游模型没有视觉能力。

不适合场景：

- 自然照片开放式理解。
- 情绪、意图、风格、物体关系、复杂场景推理。
- 没有明确结构协议的任意视觉内容。

## 6. 建议

不需要把“三个改动重新实现一遍”，因为它们已经在当前主分支。真正需要做的是：

1. 不再使用 `current_red_cluster` 这类旧输出作为当前能力判断。
2. 所有评估统一重新生成到带日期或版本号的目录。
3. 下一阶段继续做坐标校准器，目标是缩小 `depth_tolerance`，而不是只提高宽容度评分。
4. 建立更多 benchmark：表格、论文多栏、手机拍照、公式、UI 截图。
