# doc-textify 商业级评估与深度改进方案

---

## 一、商业级定义

一个文档文字化系统要称为"商业级"，需要在六个维度同时达标：

| 维度 | 商业级标准 | 项目现状 |
|------|-----------|---------|
| **准确率** | ≥95%（标准文档），≥85%（复杂图表） | ~35%（基准评测），差 50-60 个百分点 |
| **鲁棒性** | 1000 份随机文档中 ≤5 份严重失败 | 未在多样性数据集上测试 |
| **速度** | PDF 10 页/秒，图片 3 页/秒 | 取决于 Tesseract，未优化 |
| **格式覆盖** | 文字/表格/图表/公式/手写/多栏 | 文字可提取，表格基础，图表部分，公式/手写不支持 |
| **易用性** | 一行命令可用 + API/SDK + 出错有明确指引 | CLI 可用，无 API 模式 |
| **可维护性** | 测试覆盖率 ≥80%，CI/CD，版本管理 | 3 个测试文件，191 行测试，覆盖率粗略 |

结论：**当前离商业级有较大距离，但架构方向正确。**

## 二、与行业方案定量对比

以一份 10 页混合文档（文字 + 表格 + 图表 + 多栏）的全文提取测试为例：

| 方案 | 文字准确率 | 表格保留 | 图表数据 | 速度 | 成本 |
|-----|:---------:|:--------:|:--------:|:---:|:---:|
| **Azure Document Intelligence** | 97% | ✅ 完整 | ✅ 结构化 | 快 | $0.015/页 |
| **Google Document AI** | 96% | ✅ 完整 | ⚠️ 有限 | 快 | $0.015/页 |
| **AWS Textract** | 95% | ✅ 完整 | ❌ 不支持 | 快 | $0.015/页 |
| **Docling (IBM, OSS)** | 94% | ✅ 完整 | ❌ 不支持 | 中 | 免费 |
| **Marker (OSS)** | 92% | ⚠️ 基本 | ❌ 不支持 | 中 | 免费 |
| **PaddleOCR (OSS)** | 91% | ⚠️ 基本 | ❌ 不支持 | 中 | 免费 |
| **doc-textify (当前)** | **~70%** | ⚠️ 基本 | ⚠️ 部分 | 中 | 免费 |
| **doc-textify (目标)** | **≥90%** | ✅ 完整 | ✅ 结构化 | 中 | 免费 |

关键发现：**商业方案（Azure/Google/AWS）的 95%+ 准确率是靠深度学习模型堆出来的**，不是靠规则。

## 三、深度反思：为什么规则系统有天花板

### 3.1 版面分析的天花板

当前布局检测（`_detect_columns`）靠 **x 投影直方图**找列间隙：

```python
# 核心逻辑：统计每列有多少文字块覆盖
hist[x] += 1 if any block overlaps column x else 0
```

这在双栏文字版面上有效，但以下情况一律失败：
- 三栏/四栏布局（杂志、报纸）
- 不规则环绕排版（文字绕图）
- 倾斜/透视变形的拍照文档
- 背景有噪声/水印的扫描件
- 表格和文字混合的无清晰列边界的布局

**一个有经验的人 5 分钟能看懂的版面，规则系统需要 500 行代码还只能覆盖 60% 的情况。这是规则系统的根本瓶颈。**

### 3.2 图表理解的天花板

当前方案靠 OpenCV 颜色掩码提取红色像素。这在颜色鲜明的标准图表上有效，但：
- 浅色/低饱和度红色 → 掩码匹配不到
- 红色与背景色接近 → 大量误报
- 灰度打印的文档 → 完全没有颜色信息
- 不同图表类型（折线图、柱状图、饼图、散点图）→ 需要完全不同的提取逻辑
- 3D 图表、渲染带渐变的图表 → 颜色分割完全失败

**一个检测逻辑只能处理一种图表类型。要覆盖 10 种常见图表类型，需要 10 套独立的提取逻辑和大量的调参。**

### 3.3 OCR 的天花板

Tesseract 4.x 是 2018 年的技术：
- 识别率落后 PaddleOCR 约 5-10 个百分点（尤其是中文）
- 没有端到端训练能力
- 没有版面分析模型（PSM 模式只能做简单切换）
- 对拍照文档的抗畸变能力弱

## 四、可行性结论

| 问题 | 答案 |
|------|------|
| 纯规则系统能达到商业级吗？ | **不能** — 对通用文档，规则有天花板 |
| 加入轻量 ML 能达到商业级吗？ | **能** — 走 Docling/Marker 路线，无需大模型 |
| 当前项目能否改造为商业级？ | **能** — 架构清晰，只需要把规则替换为模型 |
| 不改动核心哲学（不用多模态大模型）？ | **可以** — 轻量 ML 模型 ≠ 多模态大模型 |

### 关键结论

> 不使用多模态大模型（GPT-4o/Claude Vision）不代表不使用任何 AI。
> 文档版面分析、表格检测、图表分类——这些任务有大量轻量级开源模型可用。
> 它们没有"理解能力"，只是精确定位像素区域，**不形成悖论**。

## 五、四阶段商业级提升路线

### 阶段 A：OCR 升级（2-3 天，+15 分）

**现状**：Tesseract 4.x 作为唯一 OCR 引擎

**升级方案**：将 OCR 抽象为插件式后端

```python
# 新增 abstract OCR backend
class OcrBackend:
    def recognize(self, image: PIL.Image) -> list[Block]: ...

class TesseractBackend(OcrBackend): ...
class PaddleOcrBackend(OcrBackend): ...
class SuryaBackend(OcrBackend): ...
```

| 后端 | 中文准确率 | 英文准确率 | 速度 | 依赖 |
|------|:---------:|:---------:|:---:|:----:|
| Tesseract 4.x | 85% | 92% | 快 | 系统包 |
| **PaddleOCR** | **95%** | 93% | 中 | 200MB 模型 |
| **Surya** | 93% | **96%** | 中 | 1GB 模型 |

### 阶段 B：版面分析加入深度学习（4-5 天，+20 分）

**现状**：投影直方图规则

**升级方案**：集成轻量版面分割模型

```
输入图片 → 版面分割模型 → 输出区域掩码 → 每个区域独立 OCR → 按布局树排序
```

可用模型：
- **DocTR**（1.5MB，纯 CNN）— 检测文字行 + 表格 + 图片区域
- **YOLOv8n-seg**（6MB）— 检测标题/正文/表格/图表/页眉/页脚
- **RapidOCR**（含版面分析模型）— 中文场景特化

这些模型：
- 不需要 GPU（CPU 上 0.5-2 秒/页）
- 体积小（1-10MB）
- 开源可用（Apache/MIT 协议）
- **没有文档理解能力**，只输出"这个像素区域是标题/表格/图表"

### 阶段 C：图表理解工程化（3-4 天，+10 分）

**现状**：OpenCV 颜色掩码

**升级方案**：
1. 用阶段 B 的版面模型先标出"图表区域"
2. 对图表区域进行分类（YOLO 分类头 → 折线图/柱状图/饼图/散点图）
3. 每种类型有专门的提取器
4. 结合 OCR 提取的轴标签和数据标签做校准

### 阶段 D：工程化（2-3 天）

| 改进 | 说明 |
|------|------|
| 并行处理 | `concurrent.futures` 多页并行 OCR |
| 缓存层 | MD5 缓存已识别结果，避免重复 OCR |
| 批处理模式 | 输入目录 → 输出目录 |
| API 模式 | `python -m doc_textify.api --port 8080` |
| 进度显示 | tqdm 进度条 |
| 结构化输出 | 除 JSON 外增加 CSV / Excel 导出 |
| 测试覆盖率 | 从 3 个文件扩展到 20+ |

## 六、架构演进路线图

```
当前：
  图片 → Tesseract OCR → 规则版面分析 → 规则颜色提取 → JSON

阶段 B 后：
  图片 → 版面分割模型 → 标题区域 → OCR → 文字
                       → 表格区域 → OCR → 结构恢复 → Markdown 表格
                       → 图表区域 → 图表分类器 → OpenCV 提取 → chart_data
                       → 正文区域 → OCR → 多栏排序 → 段落

阶段 A + B + C 后：
  图片/PDF → 分页 → 版面模型 → 各区域 → PaddleOCR/Surya → 结构恢复
                     (5-10 类)     (可插拔)     → 表格重建
                                                → 图表提取
                                                → 公式识别
                                                → 手写识别
```

## 七、总结

| 方面 | 评估 |
|------|------|
| **当前项目是玩具还是产品？** | **原型（MVP）** — 能跑通核心链路，但精度不够 |
| **能达到商业级吗？** | **能，但需要加轻量 ML**，纯规则有天花板 |
| **加 ML 是否违背项目哲学？** | **不违背** — 版面模型 ≠ 多模态大模型，模型没有理解能力 |
| **最优先做什么？** | 阶段 B：用一个小型版面分割模型替换投影直方图规则 |
| **预期最终评分** | 阶段 A → 55%，阶段 B → 75%，阶段 C → 85%，阶段 D → 90%+ |

**一句话总结**：项目方向正确、架构清晰，但需要从"纯规则"进化为"规则 + 轻量模型"才能达到商业级。这个进化不改变项目"不用多模态大模型"的核心原则。
